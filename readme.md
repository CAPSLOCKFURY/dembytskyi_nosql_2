# Локальний запуск проекту

### Встановлення залежностей
```bash
pip install -r requirements.txt
```

### Налаштування API-ключів
Створіть файл `.env` у кореневій директорії проекту та додайте ваш API-ключ Pinecone:
```env
PINECONE_API_KEY=your_pinecone_api_key_here
```

### Підготовка датасету
Для запуску необхідний оригінальний файл датасету `arxiv-metadata-oai-snapshot.json` (~5.2 GB), завантажений з Kaggle та поміщений у корінь проекту.

### Послідовний запуск скриптів
Усі скрипти знаходяться в папці `scripts/` та мають виконуватися строго по порядку:
1. **Попередня обробка даних**:
   ```bash
   python scripts/01_prepare_data.py
   ```
2. **Генерація ембеддингів**:
   ```bash
   python scripts/02_embed.py
   ```
3. **Завантаження векторів у Pinecone**:
   ```bash
   python scripts/03_load_to_pinecone.py
   ```
4. **Семантичний пошук та порівняння метрик**:
   ```bash
   python scripts/04_search.py
   ```
5. **Тестування стратегій чанкінгу**:
   ```bash
   python scripts/05_chunking.py
   ```
6. **Запуск гібридного пошуку**:
   ```bash
   python scripts/06_hybrid_search.py
   ```

### 2. Переваги `specter2_base` над `all-MiniLM-L6-v2` для наукових текстів
Універсальні моделі на кшталт `all-MiniLM-L6-v2` навчаються на загальних текстах з інтернету (Вікіпедія, Reddit, веб-сайти). Вони погано розпізнають складні наукові абстракції, математичну номенклатуру та специфічні взаємозв'язки між дослідженнями. 

Модель `allenai/specter2_base` спеціально розроблена **Allen Institute for AI** для наукових документів. Вона базується на архітектурі BERT і навчалася на графі наукових цитувань бази Semantic Scholar (близько 6 млн статей). 
*Цитата з HuggingFace картки моделі:*
> *"SPECTER2 is a family of models that can be used to generate document-level embeddings for scientific documents. It is trained on a large corpus of scientific papers using their citation graph to capture document-level semantic relationships."*

Модель оптимізована розуміти структуру виду `[Title] [SEP] [Abstract]` і кодувати її в єдине 768-вимірне векторне представлення. Вона вловлює не просто текстову схожість, а концептуальні наукові зв'язки (наприклад, що дві статті досліджують одну проблему різними методами, навіть якщо лексично вони не перетинаються).

### 3. Рекомендована метрика схожості
У картці моделі `specter2_base` чітко зазначено, що для порівняння векторів слід використовувати **косинусну схожість (cosine similarity)** або **скалярний добуток (dot product)** на нормалізованих векторах. 
Це критично при створенні індексу у Pinecone чи іншій базі, оскільки некоректна метрика (наприклад, звичайна Euclidean відстань без нормалізації) буде сильно спотворювати результати: вектори довших текстів або статей з більш вираженими термінами матимуть більшу довжину, що штучно висуватиме їх на перші позиції пошуку. Вибір `cosine` як метрики індексу гарантує математичну коректність пошукового простору.

## Отримання ембеддингів та математичні основи (Частина 1.3)

При підготовці даних у скрипті `02_embed.py` ми застосували прапорець `normalize_embeddings=True`.

### Теоретичне доведення еквівалентності косинусної схожості та скалярного добутку

Косинусна схожість між двома векторами $A$ та $B$ обчислюється за формулою:
$$\text{CosineSimilarity}(A, B) = \frac{A \cdot B}{\|A\|_2 \|B\|_2}$$

де:
* $A \cdot B$ — скалярний добуток (dot product) векторів,
* $\|A\|_2$ та $\|B\|_2$ — евклідові норми (довжини) векторів.

Якщо ембеддинги нормалізовані до одиничної довжини (L2-нормалізація), то їхні евклідові норми за визначенням дорівнюють одиниці:
$$\|A\|_2 = 1 \quad \text{та} \quad \|B\|_2 = 1$$

Підставимо ці значення в оригінальну формулу косинусної схожості:
$$\text{CosineSimilarity}(A, B) = \frac{A \cdot B}{1 \cdot 1} = A \cdot B = \text{DotProduct}(A, B)$$

**Висновок:** Для нормалізованих векторів косинусна схожість математично тотожна їхньому скалярному добутку. Це дозволяє здійснювати пошук значно швидше, оскільки векторним базам даних достатньо обчислити простий скалярний добуток без витратних операцій обчислення квадратних коренів для визначення довжин векторів при кожному порівнянні.

## Завантаження даних та робота з метаданими (Частина 2)

Скрипт `03_load_to_pinecone.py` здійснює завантаження 5,000 нормалізованих векторів батчами по 200 штук (рекомендований ліміт Pinecone для оптимальної швидкості).

### Чому поле `abstract` обрізається до 500 символів?
1. **Обмеження на метадані**: Pinecone встановлює жорсткий ліміт на сумарний обсяг метаданих для одного вектора (до 40 KB). Хоча окрема анотація рідко перевищує цей ліміт, при масштабуванні системи збереження надлишкових текстових полів у векторному індексі значно збільшує споживання пам'яті (RAM) та знижує швидкість фільтрації.
2. **Архітектура RAG (Retrieval-Augmented Generation)**: Векторна база даних створена виключно для швидкого пошуку ідентифікаторів (IDs) документів. Зберігати в ній великі обсяги сирого тексту — неоптимально. Правильний архітектурний підхід полягає в тому, що векторна база повертає `paper_id` та короткі метадані для швидкої візуалізації, а повний текст анотації чи статті підтягується з високопродуктивної реляційної чи NoSQL бази даних (або локального Parquet/документного сховища) за знайденим ID.

## Семантичний пошук, метафільтри та порівняння метрик (Частина 3)

Скрипт `04_search.py` тестує семантичний пошук, роботу метафільтрів та порівнює метрики схожості локально.

### 1. Результати семантичного пошуку
Запит: `"teaching machines to recognize objects in pictures"`
Знайдені статті стосуються обробки зображень, розпізнавання образів та навчання моделей на фізичних процесах (наприклад, візуалізація вузлів у полімерах, моделювання зварювальних ванн через RBF нейронні мережі та символьні сенсори).

### 2. Робота з метафільтрами
* **Фільтр А** (`category = cs.LG`, `year >= 2021`): **0 збігів**. Це повністю коректний результат, оскільки наш зріз датасету містить наукові статті виключно за **2007 рік**. 
* **Альтернативний Фільтр А** (`category = hep-ph`, `year >= 2005`): успішно повернув топ-5 статей у галузі фізики високих енергій (феноменологія), наприклад *"Physics Beyond the Standard Model and Dark Matter"* та *"Black Hole's Life at colliders"*.
* **Фільтр В** (`year < 2015`): повернув стабільний топ-5 релевантних статей, оскільки всі документи 2007 року задовольняють цю умову.

### 3. Порівняння локальних обчислень метрик (Cosine, Dot Product, L2)
За результатами консольного виведення, топ-5 документів для **Cosine Similarity**, **Dot Product** та **L2 Distance** збігаються на 100%:

1. *Чи збігаються топ-5 для Cosine та Dot Product і чому?*
   Збігаються абсолютно, оскільки вектори нормалізовані. Як доведено в Частині 1.3, для векторів одиничної довжини косинусна схожість дорівнює скалярному добутку.
2. *Чи відрізняються результати для L2 і чому?*
   Ранжування для L2-відстані абсолютно таке саме (тільки сортування йде від найменшої відстані до найбільшої). Математично евклідова відстань для одиничних векторів виражається як:
   $$\|A - B\|_2 = \sqrt{2 - 2(A \cdot B)}$$
   Оскільки відношення монотонне, мінімізація евклідової відстані $\|A - B\|_2$ призводить до тотожного результату, що й максимізація скалярного добутку $A \cdot B$.
3. *Що сталося б, якби ембеддинги не були нормалізовані?*
   Якби вектори не були нормалізовані, результати ранжування кардинально відрізнялися б між усіма трьома метриками. 
   * **Dot Product** штучно просував би в топ довші тексти або документи з високою частотою термінів (довгими векторами), навіть якщо їхній напрямок у просторі суттєво відхиляється від запиту.
   * **L2 Distance** віддавав би перевагу коротким текстам (чиї вектори лежать близько до початку координат), караючи довгі тексти за велику норму вектора.
   * **Cosine Similarity** єдина давала б стабільний релевантний результат, оскільки вона за своєю математичною природою нівелює різницю в довжині векторів, фокусуючись лише на куті між ними.

## Аналіз стратегій чанкінгу (Частина 4)

Скрипт `05_chunking.py` виділяє 30 статей з найдовшими анотаціями (довжиною ~1800 символів) та розбиває їх двома способами:

1. **Fixed-Size Chunking** (40 слів, 10 overlap) -> згенеровано **293 чанки**.
2. **Semantic Chunking** (розбиття на речення, групування до 45 слів) -> згенеровано **260 чанків**.

### Оцінка стратегій
* **Яка стратегія дає більш осмислені чанки?**
  **Semantic Chunking** дає значно більш осмислені чанки. Групування на основі цілих речень гарантує збереження логічної структури думки автора, суб'єктів та об'єктів у межах одного текстового фрагменту.
* **Чи є випадки розрізаних речень і як це впливає на ембеддинги?**
  У **Fixed-Size** чанкінгу розрізання речень посередині відбувається постійно. Це призводить до того, що частина контексту залишається в одному чанку, а завершення — в іншому. Ембеддинг такого розірваного фрагменту стає "шумним" та неповним, оскільки модель Specter2 втрачає граматичні та семантичні орієнтири (наприклад, підмет залишається в попередньому чанку, а присудок — у поточному). Це знижує якість пошуку.
* **Як розмір overlap впливає на кількість чанків і покриття тексту?**
  Збільшення overlap (перекриття) призводить до збільшення загальної кількості чанків (оскільки дані дублюються на стиках). Великий overlap покращує покриття контексту, гарантуючи, що семантичні зв'язки між сусідніми реченнями не загубляться у Fixed-Size підході. Проте занадто великий overlap веде до надлишковості векторного простору, збільшення вартості зберігання та виникнення майже ідентичних результатів у видачі.

   * **Hybrid Search** завдяки RRF вивів на перше місце *"Optimization in Gradient Networks"* (paper_1143), оскільки ця стаття була високо оцінена обома алгоритмами одночасно (4 місце в BM25 та 4 місце в Pinecone).
3. **Концептуальний запит про емоції**:
   * **BM25** зматчив точні слова "text", "human", видавши статті про методи введення тексту на комп'ютерах.
   * **Vector Search** зрозумів концепт взаємодії людини та комп'ютера і видав *"Opinion Dynamics and Sociophysics"* (динаміка емоцій/думок у соціумі) та *"Narratives within immersive technologies"*.
   * **Hybrid Search** об'єднав найкраще, поставивши на 1-ше місце *"On the Development of Text Input Method"* (який має і точні ключові слова, і семантику NLP).

### Відповіді на запитання
* **Який метод дав кращий результат і чому?**
  **Гібридний пошук** є найбільш збалансованим та надійним. BM25 ідеально знаходить точні імена чи абревіатури, але сліпий до синонімів. Векторний пошук чудово розуміє синоніми та загальну тему, але іноді ігнорує точні ключові слова. Гібридний пошук об'єднує переваги обох підходів.
* **Чи є документи в топ-5 гібридного пошуку, яких немає в топ-5 окремих методів, і чому?**
  Так, наприклад, у третьому запиті на 3-тє місце вийшов документ `paper_1266` (*"Text Line Segmentation of Historical Documents"*), якого не було в перших п'ятірках окремо. Це відбувається тому, що RRF підсумовує взаємні ранги. Якщо документ стабільно займає, наприклад, 6-те та 7-ме місця в обох методах, сумарний RRF-бал підніме його вище, ніж документи, які були лідерами (1-ше місце) в одному списку, але повністю були відсутні в іншому (ранг > 50).
* **Як зміна параметра k в RRF впливає на видачу (k=60 vs k=1)?**
  Параметр $k$ є фактором згладжування:
  * При **$k=1$** вага перших позицій є екстремально високою ($\frac{1}{1+1}=0.5$ проти $\frac{1}{1+3}=0.25$ для 3-го місця). Це дає величезний пріоритет документам, які є абсолютними лідерами хоча б в одному з методів, навіть якщо інший метод їх повністю проігнорував.
  * При **$k=60$** (класичне значення) штраф за зниження рангу розподіляється дуже плавно ($\frac{1}{61}\approx0.0164$ проти $\frac{1}{63}\approx0.0159$). Це дає перевагу документам, які мають хороші збалансовані оцінки в обох системах пошуку.

## Глибокий аналіз компромісів та архітектурні висновки (Частина 6)

### 1. Семантичний пошук vs BM25
* **BM25** виграє, коли запит містить унікальні власні назви (ім'я автора `LeCun`, назву конкретної архітектури `BERT`, `ResNet`), коди помилок, артикули товарів або точні рядкові цитати.
* **Семантичний пошук** виграє при запитах природною мовою, запитаннях ("how to train models faster") або коли користувач описує сутність явища іншими словами (наприклад, шукає "розпізнавання емоцій", а статті містять термін "sentiment analysis" або "affective computing").
* **Загальне правило**: У сучасних пошукових системах найкращим стандартом є використання **гібридного підходу** на першому етапі (retrieval) з наступним переранжуванням (cross-encoder reranking) на другому етапі.

### 2. Вплив розміру чанка
* **Занадто малий чанк (10-15 слів)**: призводить до повної втрати контексту та логіки речення. Вектор такого чанка кодує випадкові словосполучення, створюючи величезну кількість шуму та хибнопозитивних збігів у базі даних.
* **Занадто великий чанк (500+ слів)**: розмиває семантичний фокус ембеддингу (embedding dilution). Коли в один 768-вимірний вектор стискається ціла сторінка тексту з різними думками, дрібні та важливі деталі нівелюються. Також є ризик вийти за межі максимального контексту моделі (512 токенів).
* **Оптимальний розмір**: Зазвичай становить **150–250 слів** з перекриттям (overlap) у 10–20% для збереження граничного контексту.

### 3. Невідповідна метрика
Якщо створити індекс з метрикою `euclidean` (L2), але модель повертає нормалізовані вектори одиничної довжини, результати пошуку (ранжування) будуть **абсолютно правильними й ідентичними косинусній схожості**.

**Математичне обґрунтування:**
Нехай $A$ (вектор запиту) та $B$ (вектор документа) — нормалізовані вектори одиничної довжини, тобто $\|A\|_2 = 1$ та $\|B\|_2 = 1$.
Квадрат евклідової відстані між ними обчислюється як:
$$\|A - B\|_2^2 = (A - B) \cdot (A - B) = A \cdot A + B \cdot B - 2(A \cdot B)$$

Оскільки вектори нормалізовані:
$$A \cdot A = \|A\|_2^2 = 1 \quad \text{та} \quad B \cdot B = \|B\|_2^2 = 1$$

Підставимо ці значення:
$$\|A - B\|_2^2 = 1 + 1 - 2(A \cdot B) = 2 - 2(A \cdot B)$$

Оскільки косинусна схожість для одиничних векторів дорівнює їхньому скалярному добутку ($\text{CosineSimilarity}(A, B) = A \cdot B$):
$$\|A - B\|_2 = \sqrt{2 - 2 \cdot \text{CosineSimilarity}(A, B)}$$

**Висновок:** Евклідова відстань монотонно спадає зі зростанням косинусної схожості. Коли косинусна схожість максимальна (дорівнює 1), евклідова відстань мінімальна (дорівнює 0). Ранжування елементів залишиться незмінним. Проте обчислення `cosine` або `dotproduct` є ефективнішим з обчислювальної точки зору на великих масштабах.

### 4. Обмеження Pinecone Starter та масштабування на 10 мільйонів статей
* **Обмеження безкоштовного тіру**: 1 активний індекс, ліміт до 100 000 векторів розмірністю 768, відсутність підтримки розподілених просторів імен, можливий перехід індексу в сплячий режим при бездіяльності.
* **Масштабування до 10 000 000 статей**:
  1. **Перехід на self-hosted рішення**: Розгортання розподіленого кластера **Qdrant** або **Milvus** в Kubernetes.
  2. **Квантування векторів (Vector Quantization)**: Застосування Scalar Quantization (SQ8) або Product Quantization (PQ) для стиснення векторів у 4-8 разів, що критично зменшує вимоги до оперативної пам'яті (RAM).
  3. **Двохетапне індексування та шардування**: Розподіл (sharding) бази за метаданими (наприклад, за роками публікації або основними категоріями наук).
